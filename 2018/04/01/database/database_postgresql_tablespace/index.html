Postgresql表空间介绍

PostgreSQL表空间

一、什么是表空间

很多商用数据库都支持在创建数据库或表时指定数据在物理设备上的存放位置，指定的对象就叫做表空间。
这样做有很多好处：

a. 当数据库集群所在分区用满时，可以通过添加表空间来将其它分区加入到数据库集群中，从而达到扩容的目的。
b. 频繁访问的数据可以存储在性能较高的SSD上，不常用的数据存储在普通磁盘上，从而实现冷热数据的分离。
c. 将不同的数据放到不同的物理存储设备上，从而提高IO的并发度。

一句话，使用表空间的一个明显好处是能够把数据合理的分布存储在不同的设备目录下，有助于提高数据的存取效率。

二、常用表空间的管理方式

在各种数据库中常见的表空间的管理方式通常有SMS，DMS两种策略，其实就是数据文件交由操作系统管理还是数据库自行管理。

SMS（System Managed Space），表空间直接使用操作系统所提供的文件系统管理器来分配和管理表的存储空间。
在SMS表空间内，每个容器是操作系统的一个目录，表对象被创建成这个目录中的文件。
创建 SMS 表空间时，用户必须指定每个容器的目录名。通过对每个对象使用唯一的文件名，DB2 将在表空间中所使用的目录内创建表。

DMS（Database Managed Space），表空间由数据库的存储管理器来分配和管理表的存储空间。
在DMS表空间内，容器可以是操作系统文件、裸逻辑卷或裸磁盘分区。
创建DMS表空间时，用户必须指定文件、逻辑卷或磁盘分区的名称以及（一个或多个）容器的大小。

SMS DMS
对象管理 操作系统管理，一库一目录，一表一文件 数据库存储管理器分配映射表管理
空间分配 按需分配 预分配
易管理性 依赖操作系统，管理简单 管理复杂，空间大小，预取方式等需要自行控制
性能表现 很好 最好，大约可提升5%～10%的性能
应用场景 小型系统 大型商业系统

主流的商业数据库管理系统都采用了DMS方式或两种方式都支持，因为DMS方式更加灵活可以提供更好的性能表现。
DMS不论在OLAP还是OLTP系统中都比SMS的性能有不小的提升，但随着OLTP规模不断增大的情况下两者之间的差距显现其实并不大。

我们知道一个用户表主要对应三类文件：数据文件、空闲空间映射文件、可见性文件。
在DMS中，比如informix，这三类文件都存放在一起，需要按照页面的类型来区分是哪类文件，处于哪种状态。
PostgreSQL的存储引擎默认采用SMS的管理方式，即由操作系统管理数据文件，一库一目录，一表一文件。
这三类文件分开存储成三个不同的文件，例如：3600, 3600_fsm, 3600_vm。因此，这三个文件中的页面基本上都是同一类型的，当然表和索引都存储在数据文件里。

三、postgres里如何使用表空间

在PostgreSQL里表空间（tablespace）更确切的说应该叫数据存储空间，因为不管是创建数据库还是创建表都可以指定表空间。
通过\h create tablespace和\h create table我们发现他们都有指定TABLESPACE的选项。

默认initdb之后会有两个表空间pg_global和pg_default。
pg_default是用来存储系统目录对象、用户表、用户表index、和临时表、临时表index、内部临时表的默认空间，他是模板数据库template0和template1的默认表空间，它对应的物理位置为 $PGDATA/base目录。
pg_global是用来存储共享系统目录的默认空间，它对应的物理位置为 $PGDATA/global目录。

  1. pg_database是用dattablespace列来标识数据库所在的表空间。
