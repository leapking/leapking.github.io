用户态内存管理

在Linux中每个用户程序都运行在虚拟的内存空间中，操作系统内核并没有提供对虚拟内存空间的管理接口，其借助于第三方的开源实现。

开源社区公开了很多现成的虚拟内存分配器（Virtual Memory Allocators，以下简称为分配器）：
dlmalloc – 第一个被广泛使用的通用动态内存分配器；
ptmalloc2 – glibc 内置分配器的原型；
jemalloc – FreeBSD ＆ Firefox 所用分配器；
tcmalloc – Google 贡献的分配器；
libumem – Solaris 所用分配器；

Linux的早期版本采用dlmalloc作为它的默认分配器，dlmalloc不支持线程安全的多线程调用。在dlmalloc中，当两个线程同时malloc时，只有一个线程能够访问临界区（critical section）——这是因为所有线程共享用以缓存已释放内存的「空闲列表数据结构」（freelist data structure），所以使用dlmalloc的多线程应用会在malloc上耗费过多时间，从而导致整个应用性能的下降。

基于dlmalloc开发的ptmalloc2引入了多线程支持，Linux就转而采用了ptmalloc2。多线程支持可以提升分配器的性能，进而间接提升应用的性能。
在ptmalloc2中，当两个线程同时调用malloc时，内存均会得以立即分配——每个线程都维护着单独的堆，各个堆被独立的空闲列表数据结构管理，因此各个线程可以并发地从各自的空闲列表数据结构中申请内存。这种为每个线程维护独立堆与空闲列表数据结构的行为就「per thread arena」。

glibc源码: http://www.gnu.org/software/libc/libc.html

glibc手册: http://www.gnu.org/software/libc/manual/html_node/index.html

多线程内存分配介绍

「栈」指代用于分配静态内存的虚拟地址空间，在程序内存的「Stack」中通过汇编栈指令动态调整。

「堆」指代用于分配动态内存的虚拟地址空间，Glibc里通过brk系统调用调整「Heap」段大小(小于128KB的内存申请)以及通过mmap系统调用申请「Memory Mapping Segment」(超过128KB的内存申请)；

2. 主线程分配1KB内存之后

从如下的输出结果中我们可以看到，新增了主线程的heap段，并且其地址区间正好在数据段（0x09662000 - 0x09683000）上面，这表明堆内存是移动「Program Break」的位置产生的（也即通过 brk 中断）。Heap段位于低地址，map段位于高地址。

此外，请注意，尽管用户只申请了 1000 字节的内存，但是实际产生了 132KB 的堆。这个连续的堆区域被称为「arena」。因为这个 arena 是被主线程建立的，因此其被称为「main arena」。接下来的申请会继续分配这个 arena 的 132KB 中剩余的部分。当分配完毕时，它可以通过继续移动 Program Break 的位置扩容。扩容后，「top chunk」的大小也随之调整，以将这块新增的空间圈进去；相应地，arena 也可以在 top chunk 过大时缩小。

3. 主线程释放1KB内存之后，没有变化

当分配的内存区域 free 掉时，其并不会立即归还给操作系统，而仅仅是移交给了作为库函数的分配器。这块 free 掉的内存添加在了「main arenas bin」中（在 glibc malloc 中，空闲列表数据结构被称为「bin」）。随后当用户请求内存时，分配器就不再向内核申请新堆了，而是先试着各个「bin」中查找空闲内存。只有当 bin 中不存在空闲内存时，分配器才会继续向内核申请内存。当Heap段很大且brk指针位置存在连续的空闲空间时，会进行内存紧缩，使Heap段变小。

4. 主线程申请大内存之后，mmap映射区发生了变化0x9b8000到0xa36000，增加了504KB

5. 主线程释放大内存之后，大内存(>=128KB)立即归还给操作系统
